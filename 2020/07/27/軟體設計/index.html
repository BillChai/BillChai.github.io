前言

這堂課日文名為ソフトウェア工学，為一門教授商業軟體設計的SOP以及注意事項的一堂課，可以幫助自己分析以及評估出工程的發展以及好壞。

課程目標為

  1. ソフトウェア開発の流れと，大規模ソフトウェア開発の課題について理解し概要を説明できる.
  2. ソフトウェア工学の代表的なモデル，プロセス，ツールについてその概要を説明できる.
  3. ソフトウェア工学の代表的な手法を，簡単な問題に対して適用し，要求仕様書，設計仕様書，テスト仕様書等を作成できる.

第一堂課

ソフトウェア工学とは？

コンピュータ・ソフトウェアの開発，運用，保守における生産性と品質の向上を実現するための技術体系や学問体系

QCD 三個不同方向來分析

  • Q:代表QUlity 品質
  • C:代表Cost 成本

ステークホルダ(stakeholder):ソフトウェアの開発とその成果物にかかわりのあるすべての人

ステークホルダ(stakeholder)：之間的意見調整可能會要成軟體開發進程的緩慢。

デスマーチ (death march)：意指Software開發進展的疲憊以及龜速。

SLOC：プログラムの規模を測る指標として，プログラムの行数SLOCがよく使われる．

第二堂課

本課內容為開發設計流程模型的特徵以及手法的介紹

開發設計流程模型

ソフトウェア開発工程：仕事の進め方（プロセス）を理解する

プロトタイプ(prototype)：意指軟體開發之前所設計的試驗品，可幫助確認開發方向以及想發得正誤性。

  1. ウォータフォールモデル(Waterfall Model):如水流一般由上而下的作業方法。可以細分為上流工程(upper process)，下流工程(lower process)
    優點

    1. 開発の進捗管理(progress management)がしやすい
    2. 多人数の共同作業となる大規模システムの開発でも実績がある

    缺點

    1. 机上では要求事項や設計の詳細化・明確化には限界がある
    2. 要求事項も時々刻々と変化する
    3. 大量のドキュメントの作成を必要とする

  2. Ｖ字型モデル:（段階的詳細化; stepwise refinement）和（段階的統合化; stepwise integration）

  1. スパイラルモデル(spiral model)：在每一個開發階段都重複的評估開發過程以及狀況為特徵。
  2. アジャイルプロセスモデル(agile process model)：現在常見的開發手法。
    早期にリリースし，ユーザとの協力関係によりプロダクトを成長させていく方法と
    エクストリームプログラミング（ＸＰ）がアジャイルプロセスを実践する手法として有名
    1. 短期開発の反復
    2. テスト計画先行(test first)
    3. ペアプログラミング(pair programming)
    4. 連続同時レビューテスト
    5. 随時リファクタリング

要求分析與定義

要求分析與定義，為軟體開發流程中的一環，本階段著重於把客戶的要求作出分析以及實作上的評估並呈現與討論。

名詞解釋

  1. 要求:顧客的要求可分為機能要求與非機能要求。
  2. 要求仕様：要求を具体的に記述したもの。
    1. 完全性（＝モレがない; complete)
    2. 非曖昧性（＝明確; unambiguous，検証可能⇒できるだけ数値化）
    3. 無矛盾性(consistent)

第三堂課

本課內容為要求獲得中的分析手法

            graph LR
            要求獲得-->要求項目分析
要求項目分析-->要求仕樣化

要求獲得

  1. 業務フロー図（workflow diagram):組織内の役割間の関連が分かるように作業や連絡の手順・流れを表した図
  2. ゴール指向分析（目的展開）
  3. ユースケース図(Usecase Diagram):システムの機能を表すユースケースとアクターの関係を表現したもの
  4. Usecase Scenario:Scenario意為劇本，將要求的順序以及由哪個部分掌管寫清楚。
    1. アクター(Actor):シナリオに現れる登場人物．一般ユーザ，管理者など人間であることが多いが，外部システムや機械の場合もある.
    2. トリガー(Trigger):シナリオに記載される動作を起こすきっかけとなる事象
    3. シナリオ:演劇脚本のように，だれが，どういう状況で，いつ，何をすると，状況がどう変わるかを記述したもの
    4. 事後条件:シナリオの動作によって，最終的に状況がどう変化するかを表したもの

第四堂課

本課內容為使用構造化分析手法來理解以下所出現的「要求的仕樣化」內容

什麼是構造化分析？

構造化分析是一種幫助軟體開發的一種分析手法，主要的特徵為

  • 段階的詳細化
  • 図示

來幫助客戶以及分析者擁有完整以及良好的溝通效果

構造化分析的手法的提倡者：DeMarco的分析方式為

            graph LR
            A(Data Flow Diagram)-->B(Data Dictionary)
B-->C(Mini Specification)
  1. Data Flow Diagram：顧客や設計者に分かりやすくデータや処理の流れを説明
  2. Data Dictionary：データ項目の漏れや認識違いを防ぐ
  3. Mini Specification：処理手続きを文章や疑似プログラムで表現

要求的仕樣化

  1. データフローモデル(data flow model):対象物（データ）を処理する順序を図示する手法
  2. データ辞書:データ項目が他のデータや値からどのように構成されるかを言語生成規則のように表現する手法
  3. 状態遷移モデル(state transition model):「状態」をノードとし，「事象（イベント）」による「状態の遷移」をノードとノード間とエッジで表したもの
  4. 決定表（ディシジョンテーブル）:複雑な条件を整理して，行動を記述するための表
  5. 実体関連モデル(entity relationship model; ERmodel):実体（エンティティ）間の関係を図示したもの

第五堂課

本課內容為物件導向的仕樣化分析

什麼是物件導向分析？

実世界のモ
デルをソフトウェアで表現するために「オブジェクト」を構
成単位としてソフトウェアを構築する枠組み.

構造化分析 オブジェクト指向分析
機能中心 オブジェクト中心
処理（プロセス）へデータが流れてくる オブジェクトへメッセージがくる
データフロー図 クラス図，オブジェクト図など
手続き型言語: COBOL, PL/I, C言語 オブジェクト指向言語 ：Java,C++

使用物件導向的好處

  1. 手続き型：プログラマーがデータ型を意識して，適切な手続き（関数）を選ぶ必要がある.
  2. オブジェクト指向：オブジェクトが自分自身のデータの操作方法を知っているので，メッセージを送るだけで良い.

物件導向的基本概念

  1. メッセージパッシング:機能を呼び出すのではなく，オブジェクトにメッセージを送ることで処理を依頼する考え方
  2. カプセル化 : データ（属性）とそれに特化した操作（メソッド）をまとめてモジュール化すること
  3. クラスとインスタンス:データの構造と操作が同じインスタンスを簡単に作る仕組み
  4. 継承 : クラスを階層化し，親クラス（上位クラス）のもつ属性操作を子クラス（下位クラス）の属性・操作として共有することで冗長性を削減する仕組み

フレームワーク
  1. 概略設計（アーキテクチャ設計）：ソフトウェアの全体構造と組立てを記述する
  2. 詳細設計： 各モジュールの振る舞いやインタフェースを記述する

  4. パターン設計： 複数の対象設計で共通的に使えるモジュールやその組み合わせ方を設計するメタレベルの設計活動.

要求割り付け

要求割り付け(requirements allocation): 個々の要求を満足さ
せるために，要求をサブシステムに割り付けること.

ソフトウェアアーキテクチャ

ソフトウェアアーキテクチャ：複数の対象設計で共通的に用
いることができる構造と組み合わせ方を類型化（パターン
化）したもの.

  1. 論理ビュー（モジュール構造）:モジュールの分割方法と相互作用
    （１）階層モデル：三層モデル，仮想マシンモデル
    （２）MVCモデル
  2. 実行ビュー（プロセス構造） ･･･プロセスの分割方法と通信方法
    （３）データフローモデル：バッチ処理，パイプ・フィルタ
    （４）コントロールモデル：集中型，イベント駆動型
  3. 配置ビュー（プロセス配置）･･･プロセスのコンピュータへの配置方法
    （５）クライアントサーバモデル，集中処理，水平分散処理

第八堂課

本章節著重於介紹第七堂課介紹的詳細設計（Detailed Design）的部分，通過本堂課希望能夠理解軟體設計的Module的評價方法以及優劣

モジュール設計とは

モジュール設計：ソフトウェアを構成部品に分割し，構造化する作業.

モジュール化の利点

  1. モジュール単位で抽象化することで，部品を組み合わせるようにソフトウェアを構成できる
  2. モジュール単位で独立しているので並列開発，再利用可能

構造化設計

データの流れに着目した分割技法

  1. STS分割(Source Transform Sink Decomposition):データが変換される箇所（最小抽象入力点,最大抽象点という）を見つけて，入力処理，変換処理，出力処理に３分割し，それらを制御するモジュールを主モジュールとして配置する方法
  2. TR分割(Transaction Decomposition):トランザクションを入力するモジュール，属性毎に振り分けるモジュール，トランザクションごとの処理モジュールに分割
  3. 共通機能分割:STS分割，TR分割などで分割されたモジュールの中に共通する機能を持ったモジュールがある場合，それを共通モジュールとして独立させる

データの構造に着目した分割技法

  1. ジャクソン法(JSP: Jackson Structured Programming)
  2. ワーニエ法(Warnier method)

モジュール分割の評価

  1. モジュール強度：モジュール内の関連性の強さ
  2. モジュール結合度：モジュール間の関連性の強さ

デザインパタン

デザインパターン：様々なプログラムで再利用できる汎用的な設計に
名前をつけカタログ化したもの（自分でコーディングが必要）

構造化プログラミング(structured programming)

構造化プログラミング＝プログラムを構造的に書く上での指針

  1. 制御構造 (control structure)
    1. GOTO文を際限なく使うとスパゲティプログラムになるので避けよう
    2. 逐次，選択，反復の3種類のブロック（＝構造を組み立てる要素）で構成すると，ブロックの入り口と出口が一つになり，上から下に読めるから読みやすい
  2. データ構造 (data structure)
    1. 全てのデータを一次元的に並べるだけだと解析しづらいので避けよう
    2. 配列，構造体を使って構成する
  3. 段階的詳細化（stepwise refinement）
    いきなり細かい処理の記述をせずに，プログラムの処理概要から段階的に詳細化していこう

構造化定理為使用逐次（順次，順接），選択（分岐），反復三種構造組成的。

Code的可讀性(coding standard/convention)

目的：可読性や保守性を高める

規約の概要:

  1. 変数や関数の命名規則
    • 分かり易い名前にする
    • キャメルケース(getFileName())，スネークケース(get_file_name())
  2. コーディングの規則
    • コメントの記載内容
    • 空白文字，インデント，改行の入れ方
  3. 禁止事項
    • 大域変数やgoto文の禁止
  4. その他
    • 著作権表記

開發環境的介紹

統合開発環境(IDE; Integrated Development Environment)

擁有以下三個部分

  1. Editor
  2. Debug,性能解析ツール（プロファイラ）
  3. Compiler
  4. 版本管理或小組管理

高級語言

  1. 手続き型プログラミング：コンピュータの処理手順を命令文で記述する
    • 例：FORTRAN, COBOL, BASIC, PASCAL, C, Ada, Java
  2. 関数型プログラミング：複数の関数の組み合わせ（合成）によって処理を記述する
    • 例：Lisp, Scheme, Haskell
  3. 論理型プログラミング：入出力関係を述語論理（事実と規則）で記述す
    • 例：Prolog

第十堂課

本章節著重於介紹軟體開發時針對開發狀況以及開發過程的評估方式

ソフトウェアの品質管理では**欠陥はなるべく早期に除去するのが鉄則。**そのために、プログラムの検査を行い、単体テスト、システムテストや受入検査などの方法がある。

ソフトウェアテスト:正しいだけじゃなく、欠陥を見つ けるためにテストを行う。

テストの種類

  1. 単体テスト:単一のプログラムに対してテストする。
  2. 結合テスト:複数なプログラムに対して、正しく相互作用しているかをテストする。
  3. システムテスト:システム全体の振る舞いをテストする。

単体テスト

  1. White Box Test:内部仕様、Source Codeなど
  2. Black Box Test:外部仕様。

テスト方法:

  1. 制御パステスト法:
    1. 命令Coverage(C0 Coverage):すべての制御Flow Graphのノード(命令文 ) を網羅。
    2. 分岐Coverage(C1 Coverage):命令網羅と命令網羅ではカバーされない制御フローグ ラフのアークを網羅。
    3. Path Coverage:Pathごとにテストするが、なかなか量が多きので。
  2. 同値分割法:有効な値の集合と無効な値の集合をテストする。
  3. 境界値分析法:値の境界の左右をそれぞれテストすること。
  4. Driver: Define some parameter and test one of the function or program
  5. Stub: Test the Demi Module by the forward Module.

xUnitでは自動的にテストを実行するFrameWork

Difficulties:
It’s hard to detect all bugs by the coverage way

  • Loop bug is hard to detect
  • Hard to detect bugs when the document was not detail enough - Some special Time Bugs

第十ㄧ堂課

本章節著重於介紹軟體開發時品質的評估以及希望到達的品質特徵。

結合テスト項目表：設計仕様書に記載された仕様どおりにプログラムが動作することを確認。

状態遷移テスト：状態遷移マトリックスのとおりにソフトウェアが動作しているかチェック。

信頼度成長曲線和欠陥分析

信頼度成長曲線：テストの進捗（時間経過）を横軸に，検出したバグの数を縦軸にしたグラフ（バグ管理図，バグ曲線などとも呼ぶ）。

  1. バグ検出数が目標値を大きく上回る場合
    • モジュールの品質が非常に低い可能性
    • 品質の悪いモジュールのレビューや単体テストからやり直す
    • 予測値を見直し
  2. バグ検出数が目標値を大きく下回る場合
    • テストケースが不適切である可能性
    • テストケースの見直し
    • 予測値の見直し
  3. バグ検出数が収束していない場合
    • モジュールの品質が非常に低い可能性
    • 品質の悪いモジュールのレビューやテストをやり直す
    • 目標値を見直し
  4. 信頼度成長曲線が収束傾向にあり，バグ検出数の実績と目標値の差が小さい
    • 作り込んだバグを十分に摘出した
    • テスト終了

ソフトウェア品質特性

  • 製品品質と利用品質
  • 機能性，信頼性，使用性，効率性，保守性，可搬性

本章節引用PMBOK (project management body of knowledge)方面的想法以及評估方式

時間管理（タイムマネジメント）

目的：プロジェクトを所定の時期に完了させること
ガントチャート(Gantt chart)：プロジェクトを管理するために、作業項目を木構造で階層を表示し、全体の作
業の流れおよび進捗状況を表したもの

  • 計画
  • 進捗管理
  • WBS(work breakdown structure): プロジェクトを作業項目（タスク）に分解

PERT図の作成方法

  1. ＷＢＳの作成
  2. タスクの所要時間を見積もる
  3. タスクの順序関係を設定する
  4. PERT図（アローダイアグラム）を作成する
  5. 最短所要時間を算出する
  6. 重点作業項目の洗い出し

進捗管理

進捗管理：プロセスの実行（実働）が計画に沿っているかを監視（モニタリング）し，計画からのずれが小さくなるよう制御すること
